# Perplexity

1. С чего начать диагностику
Сделай на медленном сервере:

Включи лог медленных запросов:

sql
ALTER SYSTEM SET log_min_duration_statement = '1000ms';
ALTER SYSTEM SET log_statement = 'none';
SELECT pg_reload_conf();
Потом запусти отчёт и посмотри в postgresql.log, какие именно запросы бегут и сколько длятся.

Для самого тяжёлого запроса сними план:

sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE) <текст_запроса>;
То же самое сделай на быстром сервере и сравни планы: типы join’ов, использование индексов, количество строк по оценке vs факту, seq scan vs index scan.

Чаще всего окажется, что:

На медленном сервере planner выбрал другой план (например, nested loop по большой таблице, seq scan вместо index scan).

Либо статистика устарела и оценка cardinality кривая.

2. Что проверить по настройкам
На медленном PostgreSQL 15:

Автоматическое обслуживание:

sql
SHOW autovacuum;
SHOW autovacuum_analyze_scale_factor;
SHOW autovacuum_vacuum_scale_factor;
Если autovacuum не успевает — таблицы раздуваются, статистика устаревает.

Планировщик:

sql
SHOW random_page_cost;
SHOW seq_page_cost;
SHOW effective_cache_size;
SHOW work_mem;
SHOW maintenance_work_mem;
На разных серверах эти параметры могут сильно отличаться, отсюда разные планы.

Настройки для больших сортировок и join’ов:

work_mem слишком маленький → всё пишет на диск, дикий тормоз.

cpu_tuple_cost, cpu_index_tuple_cost могут влиять на выбор планов.

Очень полезно взять SHOW ALL; на обоих серверах и diff’нуть.

3. Индексы и статистика
Для ключевых таблиц, которые участвуют в отчёте:

Проверить наличие/совпадение индексов:

sql
\d+ schema.table
Обновить статистику:

sql
ANALYZE schema.table;
VACUUM (ANALYZE) schema.table;
Проверить n_live_tup, n_dead_tup в pg_stat_user_tables — если n_dead_tup большое, autovacuum не справляется, это влияет на планы и I/O.

4. Железо и окружение
Сравни:

Размер RAM, тип диска (SSD/HDD, локальный/LUN), наличие contention по I/O (iostat, sar).

Ограничения по CPU (cgroups, контейнеры, overcommit).

Параметры shared_buffers, effective_cache_size, max_connections.

Даже если версии 13 и 15 сами по себе плюс‑минус одинаковы по перфу, криво настроенный 15‑й легко окажется в десятки раз медленнее.

5. Что включить для более детального логирования
На медленном сервере:

sql
ALTER SYSTEM SET log_checkpoints = on;
ALTER SYSTEM SET log_lock_waits = on;
ALTER SYSTEM SET deadlock_timeout = '1s';
ALTER SYSTEM SET log_temp_files = '0';  -- логировать все временные файлы
SELECT pg_reload_conf();
После выполнения отчёта посмотри в логе:

Не льются ли гигантские temp‑файлы (значит мало work_mem / плохие планы).

Нет ли долгих блокировок.

Нет ли частых checkpoint’ов, которые душат I/O.

6. Практический чек‑лист
Включить log_min_duration_statement и прогнать отчёт → взять самый тяжёлый SQL.

EXPLAIN (ANALYZE, BUFFERS) этого SQL на обеих БД → сравнить планы.

Проверить/сравнить:

индексы;

статистику (ANALYZE);

work_mem, random_page_cost, effective_cache_size, shared_buffers.

Проверить I/O (нет ли свопа, медленного диска) и autovacuum.

Если хочешь, можешь сюда скинуть EXPLAIN (ANALYZE, BUFFERS) одного конкретного запроса с обеих серверов — можно будет предметно показать, куда именно он упирается.

Шаг 1. Включите и проанализируйте логи PostgreSQL
1.1. Настройте логирование в postgresql.conf на обоих серверах:

ini
log_statement = 'all'           # логировать все запросы
log_min_duration_statement = 1000  # запросы дольше 1 с
log_checkpoints = on
log_connections = on
log_disconnections = on
log_lock_waits = on
log_temp_files = 0             # логировать все временные файлы
log_autovacuum_min_duration = 0
1.2. Найдите логи:

обычно в /var/log/postgresql/;

или в каталоге данных PostgreSQL (pg_log/);

путь можно уточнить командой:

sql
SHOW log_directory;
1.3. Ищите в логах:

долгие запросы (duration > 120 000 ms);

ожидания блокировок (lock timeout);

использование временных файлов (индикатор нехватки work_mem);

автовакуум, мешающий выполнению запросов.

Шаг 2. Сравните конфигурации PostgreSQL
2.1. Ключевые параметры для сравнения:

shared_buffers — обычно 25–40 % от ОЗУ;

work_mem — память на операцию сортировки/хеширования;

maintenance_work_mem — для операций обслуживания;

effective_cache_size — оценка доступной памяти для планирования запросов;

random_page_cost — стоимость случайного чтения (зависит от типа дисков);

max_parallel_workers_per_gather — параллелизм;

wal_level, checkpoint_timeout и др.

2.2. Как получить текущие значения:

sql
SELECT name, setting, unit, context
FROM pg_settings
WHERE name IN (
  'shared_buffers', 'work_mem', 'maintenance_work_mem',
  'effective_cache_size', 'random_page_cost',
  'max_parallel_workers_per_gather'
);
Шаг 3. Проанализируйте план выполнения запроса
3.1. Получите план на обоих серверах:

sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
-- ваш запрос для формирования отчёта
3.2. Что сравнивать в планах:

алгоритмы соединений (Nested Loop vs Hash Join vs Merge Join);

типы сканирований (Seq Scan vs Index Scan vs Bitmap Scan);

оценки кардинальности (ожидаемое vs фактическое число строк);

использование индексов — есть ли нужные индексы?

параллельное выполнение — используется ли?

стоимость операций — где «узкие места»?

3.3. Инструменты визуализации планов:

 — анализ текстового вывода;

pgMustard — десктоп‑приложение для анализа;

встроенные плагины в DBeaver, DataGrip и т. д.

Шаг 4. Проверьте статистику и индексы
4.1. Убедитесь, что статистика актуальна:

sql
ANALYZE;  -- обновить статистику для всех таблиц
Или для конкретной таблицы:

sql
ANALYZE your_table_name;
4.2. Проверьте фрагментацию индексов:

sql
SELECT schemaname, tablename, indexname,
       pg_stat_get_dead_tuples(indrelid) AS dead_tuples
FROM pg_indexes
WHERE tablename = 'your_table';
При высокой фрагментации — перестройте индексы:

sql
REINDEX TABLE your_table;
Шаг 5. Мониторинг системных ресурсов во время выполнения
5.1. Запустите мониторинг на обоих серверах одновременно:

CPU: top, htop, mpstat -P ALL 1;

память: free -h, vmstat 1;

диски: iostat -x 1, iotop;

сеть: iftop, nethogs.

5.2. Обратите внимание на:

загрузку CPU — близка ли к 100 %?

своппинг — если идёт, значит не хватает work_mem;

дисковый I/O — высокая задержка (await в iostat) или очередь (aqu-sz);

пропускную способность сети — если данные передаются между серверами.

Шаг 6. Проверьте данные и окружение
6.1. Сравните:

объём данных в таблицах — может, на сервере с PostgreSQL 15 их в 10 раз больше?

распределение данных — неравномерность может ломать планы запросов;

версии и настройки 1C — разные параметры могут генерировать разные SQL.

6.2. Убедитесь, что:

схемы БД идентичны (таблицы, индексы, представления);

данные сопоставимы по объёму и структуре;

нет блокировок от других процессов.

Шаг 7. Дополнительные инструменты диагностики
7.1. Расширение pg_stat_statements

Включите в postgresql.conf:

ini
shared_preload_libraries = 'pg_stat_statements'
pg_stat_statements.track = all
После перезапуска:

sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
Затем:

sql
SELECT * FROM pg_stat_statements
ORDER BY total_time DESC
LIMIT 10;
Покажет топ‑10 самых долгих запросов.

7.2. auto_explain — автоматический сбор планов долгих запросов

В postgresql.conf:

ini
session_preload_libraries = 'auto_explain'
auto_explain.log_min_duration = 5000  -- логировать планы запросов > 5 с
auto_explain.log_analyze = true
Краткий план действий
На обоих серверах: включите логирование (log_min_duration_statement), соберите логи во время формирования отчёта.

Сравните postgresql.conf — ключевые параметры памяти, параллелизма, I/O.

Получите и сравните планы выполнения (EXPLAIN ANALYZE) для запроса отчёта. Ищите различия в алгоритмах и оценках.

Проверьте статистику и индексы — обновите статистику, перестройте фрагментированные индексы.

Запустите системный мониторинг (iostat, top) во время выполнения отчёта — сравните загрузку CPU, I/O, памяти.

Убедитесь, что данные сопоставимы — объём, распределение, блокировки.

Используйте pg_stat_statements и auto_explain для автоматической диагностики.

Типичные причины разницы в производительности
Нехватка work_mem → сортировка на диске (временные файлы) вместо памяти.

Устаревшая статистика → плохой план запроса (например, Nested Loop вместо Hash Join).

Разные алгоритмы соединения из‑за настроек или статистики.

Отсутствие нужных индексов на сервере с медленным выполнением.

Высокая фрагментация индексов → замедление Index Scan.

Дисковый I/O — медленный диск, высокая задержка чтения.

Параллелизм — в PostgreSQL 13+ улучшены механизмы параллельного выполнения.

Итог
Начните с:

EXPLAIN (ANALYZE) на обоих серверах — это сразу покажет, где и почему запрос тормозит.

Логи PostgreSQL с log_min_duration_statement — зафиксируйте все долгие операции.

Мониторинг I/O и памяти (iostat, vmstat) — исключите аппаратные ограничения.

Скорее всего, разница вызвана разными планами выполнения из‑за:

устаревшей статистики на сервере с PostgreSQL 15;

неоптимальной конфигурации (work_mem, shared_buffers);

отсутствия нужных индексов.

Шаг 1: Проверка аппаратных ресурсов

Проверьте конфигурацию оборудования обоих серверов — различия в объеме оперативной памяти, процессорах и дисковых подсистемах могут сильно влиять на производительность запросов. Для проверки загрузки сервера используйте команды top, htop или мониторинговые утилиты вроде sar.

Bash

free -h     # проверить объем доступной RAM
vmstat      # загрузка CPU, свопинга и IO
iostat      # нагрузка на диски
▌ 🔍 Шаг 2: Конфигурационные файлы

Версии PostgreSQL имеют разные настройки по умолчанию. Например, размер буферов (shared_buffers), количество рабочих процессов (max_worker_processes) и настройка операций ввода-вывода (effective_io_concurrency) могли изменяться между версией 11 и 15. Проверьте файл конфигурации /etc/postgresql/<версия>/main/postgresql.conf на обеих системах и сравните ключевые параметры.

Обратите особое внимание на следующие переменные:

- shared\_buffers — общий объём выделенной памяти для обработки данных
- work\_mem — память для сортировок и хеширования
- maintenance\_work\_mem — память для служебных операций (например, создание индексов)
- effective\_cache\_size — предполагаемый объём кешируемого ОС кеша
- random\_page\_cost — стоимость случайного чтения страницы
- enable\_bitmapscan, enable\_hashjoin, enable\_mergejoin, etc. — включение определенных методов оптимизации запросов

▌ 🔍 Шаг 3: Статистика и индексирование

Убедитесь, что таблицы правильно проиндексированы и статистика собранная вовремя обновляется. Часто устаревшая информация о распределении данных замедляет запросы. Вы можете вручную обновить статистику командой:

SQL

ANALYZE your_table_name;
Также проверьте наличие полезных индексов для вашего отчета. Это легко сделать с помощью EXPLAIN ANALYZE:

SQL

EXPLAIN ANALYZE SELECT * FROM your_report_query;
Посмотрите разницу планов исполнения запроса на двух серверах — возможно, оптимизатор выбрал совершенно разные пути выполнения.

▌ 🔍 Шаг 4: Параметры журналирования

PostgreSQL позволяет детально регистрировать активность базы данных, включая медленные запросы и выполнение отдельных этапов выполнения SQL-запросов. Настройте ведение журнала ошибок и предупреждений:

Откройте файл конфигурации postgresql.conf и установите следующие значения:

ini

log_min_duration_statement = 1000   # Записывать все запросы дольше 1 секунды
log_checkpoints = on                # Журнал контрольных точек
log_connections = on                 # Логин пользователей
log_disconnections = on              # Выход пользователей
log_lock_waits = on                  # Журналирование блокировок
log_autovacuum_min_duration = 0     # Подробный журнал автоочистки
log_line_prefix='%m [%p]: [%l-1] user=%u,db=%d,app=%a,client=%h '
Перезапустите PostgreSQL после изменения настроек.

Затем просматривайте журналы, расположенные в каталоге /var/log/postgresql. Обратите внимание на предупреждения, длительные операции и возможные блокировки.

▌ 🔍 Шаг 5: Анализ производительности

Используйте инструменты мониторинга производительности PostgreSQL:

- pg\_stat\_statements: расширение, которое собирает статистику по всем выполняемым запросам. Установите его, активируйте и изучите самые долгие запросы.
- pgBadger: инструмент анализа журналов PostgreSQL, создающий отчёты по работе базы данных.
- explain.depesz.com: веб-сервис для визуализации плана выполнения запроса (необходимо скопировать вывод EXPLAIN).
